NOUS Language Reference

v4.0.0 · Last updated 14 April 2026

NOUS (Νοῦς — Greek for "mind/intellect") is a self-evolving programming language purpose-built for agentic AI systems. It is not a framework, not a library, not a wrapper. It is a complete language with its own grammar, parser, type system, compiler, runtime, and IDE.

The core insight: AI agents are not functions. They are living systems with memory, instincts, senses, healing ability, and evolution. NOUS models them as such.

NOUS lets you describe multi-agent AI systems in a declarative, human-readable syntax. You define Souls (autonomous agents), Messages (typed data contracts), a Nervous System (routing between souls), and World Laws (constraints like cost ceilings). The compiler generates production-ready Python or JavaScript.

# Clone the repository
$ git clone https://github.com/contrario/nous.git

First Program

Create a file hello.nous:

world HelloWorld {
    law cost_ceiling = $0.10 per cycle
    heartbeat = 30s
}

message Greeting {
    text: string
}

soul Greeter {
    mind: claude-sonnet @ Tier0A
    senses: [http_get]
    memory {
        count: int = 0
    }
    instinct {
        remember count += 1
        speak Greeting(text: "Hello from NOUS!")
    }
    heal {
        on timeout => retry(3, exponential)
    }
}

nervous_system {
    Greeter -> Greeter::Greeting
}

Then compile, verify, and run:

$ python cli.py compile hello.nous
$ python cli.py verify hello.nous
$ python cli.py run hello.nous

World

A world defines the environment in which souls operate. It sets global constraints and parameters.

world TradingPipeline {
    law cost_ceiling = $0.50 per cycle
    law max_retries = 5
    heartbeat = 30s
    timezone = "UTC"
}

Laws

Laws are constraints enforced at compile-time and runtime. cost_ceiling sets the maximum LLM spend per cycle — the circuit breaker trips automatically if exceeded. max_retries caps retry attempts in heal blocks.

Heartbeat

The heartbeat interval determines how often heartbeat souls wake up. Supports: s (seconds), m (minutes), h (hours).

Soul

A soul is an autonomous agent. It is the fundamental unit of computation in NOUS.

soul Scanner {
    mind: claude-sonnet @ Tier0A
    senses: [http_get, http_post]

    memory {
        scan_count: int = 0
        last_result: string = ""
    }

    instinct {
        let raw = sense http_get(url: "https://api.example.com/data")
        remember scan_count += 1
        remember last_result = raw
        speak Signal(source: "scanner", data: raw)
    }

    heal {
        on timeout => retry(3, exponential)
        on api_error => hibernate(60s)
    }

    dna {
        threshold: 0.7 ~ [0.1, 0.99]
        batch_size: 10 ~ [1, 100]
    }
}

Mind

The mind assigns an LLM with a cost tier: Tier0A (cheapest), Tier0B, Tier1A, Tier1B (most expensive). The cost tracker uses these tiers to estimate per-cycle cost.

Senses

Senses are external tools the soul can invoke: http_get, http_post, file_read, file_write, or custom plugin-registered tools.

Heal

Error recovery strategies: retry(n, strategy) with exponential or linear backoff; hibernate(duration) to pause; fallback(soul) to delegate; alert(channel) to notify.

DNA

Evolvable parameters with mutation ranges. The evolution engine (nous evolve) mutates DNA values within specified bounds to optimize performance metrics.

Message

Messages are typed data contracts for inter-soul communication.

message Signal {
    source: string
    data: string
    confidence: float = 0.0
    tags: [string] = []
    metadata: {string: string}?
}

Fields can have default values. Optional fields use type? syntax. The type checker validates all speak/listen calls against message definitions.

Nervous System

The nervous system defines message routing topology.

nervous_system {
    Scanner -> Analyzer::Signal
    Analyzer -> Executor::Decision
    Executor -> Scanner::Feedback  # feedback loop
}

Routes determine wake strategies automatically. Souls with no incoming routes become HEARTBEAT souls. Souls with incoming routes become LISTENER souls.

Deadlock prevention: The verifier checks for circular dependencies (excluding explicit feedback loops) and ensures no soul listens to itself.

Event-Driven Runtime

The NOUS runtime is built on asyncio. It is a true event-driven system:

  • SoulRunner — manages a soul's lifecycle. Heartbeat souls sleep between cycles. Listeners block on channel receive.
  • ChannelRegistry — named async channels with backpressure (max queue 100).
  • CostTracker — shared across all souls. Trips circuit breaker if cost exceeds ceiling.
  • SenseCache — deduplicates identical sense calls within a single instinct execution.
  • SenseExecutor — routes sense calls to registered tool functions or HTTP endpoints.

Distributed Runtime

For multi-machine deployment, NOUS adds TCP-based channel transport:

  • ChannelServer — asyncio.start_server listening for JSON-line protocol messages
  • RemoteChannelClient — auto-reconnect with exponential backoff (max 10 attempts)
  • DistributedChannelRegistry — routes to local queues or remote nodes based on topology

Envelope protocol: newline-delimited JSON:

{"op":"send","channel":"Scanner_Signal","data":{...},"node":"alpha"}\n

Topology

topology {
    node_alpha: "192.168.1.10:9100" {
        souls: [Scanner]
    }
    node_beta: "192.168.1.20:9100" {
        souls: [Analyzer, Executor]
    }
}

Node selection: --node node_alpha flag, NOUS_NODE env var, or no node for local-only runtime.

Architecture Diff

The nous diff command performs semantic impact analysis when you change a .nous file. Unlike text-based diffs (git diff), NOUS understands your agent architecture and reports meaningful behavioral changes — not line-by-line character differences.

Usage

$ nous diff original.nous modified.nous
$ nous diff original.nous modified.nous --json

What It Analyzes

The diff engine inspects 12 categories of change across two versions of a program:

Topology        Souls added, removed, or modified
Routes          Nervous system route changes
Protocol        Message types added or removed
Cost            Per-soul cost delta, total change, daily/monthly projections
Deadlock        New circular dependencies introduced
Liveness        Lost entrypoints, unreachable souls
Reachability    Souls that become unreachable
Wake Strategy   HEARTBEAT / LISTENER changes
Capability      Senses added or removed per soul
Memory          Memory fields added or removed
Evolution       DNA genes added or removed

Severity Levels

CRITICAL  Breaking changes. Deployment may fail or behave incorrectly.
WARNING   Risky changes. Review before deploying.
INFO      Safe changes. No action required.

Soul Mitosis

Self-replicating agents with formal verification gate. When a soul detects overload, it spawns a clone of itself. The clone is formally verified before deployment, and the nervous system rewires automatically. No other framework has this.

Syntax

soul Scanner {
    mind: claude-sonnet @ Tier0A
    mitosis {
        trigger: scan_count > 50 || queue_depth > 10
        max_clones: 3
        cooldown: 60s
        clone_tier: Groq
        verify: true
    }
}

Fields

trigger          Expression using memory fields + built-in metrics
                 Built-in: cycle_count, queue_depth, latency, error_count
max_clones       Maximum clones per soul (default: 3)
cooldown         Minimum time between mitosis events (default: 60s)
clone_tier       LLM tier for clones (default: parent tier)
verify           Run formal verification before deploy (default: true)

How It Works

Soul instinct runs → metrics updated
  ↓
MitosisEngine checks trigger condition
  ↓ triggered + cooldown passed + max not reached
Build temporary NousProgram with clone
  ↓
Run NousVerifier (resource bounds, deadlock, protocol)
  ↓ PASS
Deploy clone as new SoulRunner
  ↓
Listener clones share input channel (load balancing)

Validation Rules

MT001  ERROR   max_clones must be >= 1
MT002  WARN    max_clones > 10 is very high
MT003  ERROR   trigger condition is required
MT004  ERROR   soul with mitosis must have a mind
MT005  WARN    heal block recommended for mitosis souls
MT006  ERROR   clone_tier must be a valid tier

Verification Proofs

VMI001  Worst-case cost (parent + clones) ≤ ceiling
VMI002  Verification gate enabled/disabled status
VMI003  Clone tier cost vs parent tier
VMI004  Listener clones share channel (load balancing)
VMI005  Total mitosis capacity across system

CLI

$ nous verify file.nous     # Includes VMI001-VMI005 proofs

Immune System

Adaptive error recovery with LLM-generated antibodies. When heal fails, the immune system intercepts the error, generates a dynamic Python patch via the soul's LLM, caches it, and broadcasts to all clones. Defense layers: heal (static) → immune (adaptive) → death.

Syntax

soul Collector {
    mind: claude-sonnet @ Tier0A
    immune_system {
        adaptive_recovery: true
        share_with_clones: true
        antibody_lifespan: 3600s
    }
}

Fields

adaptive_recovery    Enable LLM-generated fallback functions (default: true)
share_with_clones    Broadcast antibodies to mitosis clones (default: true)
antibody_lifespan    How long antibodies remain active (default: 3600s)

How It Works

Error occurs in soul instinct
  ↓
[1] heal {} — static rules (retry, fallback, hibernate)
  ↓ heal fails
[2] Check AntibodyCache — signature match?
  ↓ YES → apply cached fallback → done (zero LLM cost)
  ↓ NO
[3] Try 7 builtin antibodies (DNS, timeout, rate limit, auth, etc.)
  ↓ no match
[4] LLM generates async fallback function
  ↓
[5] Sanitize code (strip imports, eval, exec, os, sys)
[6] Compile in restricted sandbox
[7] Execute with 5s timeout
  ↓ SUCCESS
[8] Cache antibody (with lifespan + effectiveness tracking)
[9] Broadcast to all clones via MitosisEngine clone groups

Builtin Antibodies

DNS resolution errors       (no address, getaddrinfo)
Timeout errors              (timed out, read timeout)
Connection errors           (refused, reset)
Rate limiting               (429, too many requests)
Authentication errors       (401, 403, unauthorized)
Server errors               (500, 502, 503, 504)
Parse errors                (json, decode, invalid)

Security

Generated antibodies execute in a restricted sandbox: limited __builtins__ (no open, os, sys, subprocess, eval, exec). Code sanitization strips dangerous imports. 5-second timeout on execution. Ineffective antibodies auto-removed when success rate drops below 50%.

Validation Rules

IM001  WARN    immune_system without heal block
IM002  WARN    share_with_clones without mitosis
IM003  ERROR   antibody_lifespan too short (< 10s)

Verification Proofs

VIM001  Adaptive recovery enabled/disabled
VIM002  Clone broadcast: antibodies reach N clones
VIM003  Antibody lifespan within safe bounds (< 24h)
VIM004  Layered defense: heal (static) + immune (adaptive)
VIM005  Immune coverage: N/M souls protected

Agent Dreaming

Speculative pre-computation during idle time. When a soul has no work, it enters a REM state and uses a cheap dream_mind LLM to pre-compute likely future scenarios. On wake, a check_dream_cache sense provides instant access to pre-computed insights.

Syntax

soul Oracle {
    mind: claude-sonnet @ Tier0A
    senses: [check_dream_cache]
    dream_system {
        enabled: true
        trigger_idle_sec: 30
        dream_mind: llama-8b @ Cerebras
        max_cache: 20
        speculation_depth: 3
    }
    instinct {
        let cached = sense check_dream_cache(query: "optimization")
        if cached != "" {
            # Use pre-computed insight — zero LLM cost
        }
    }
}

Fields

enabled             Enable/disable dreaming (default: true)
trigger_idle_sec    Seconds idle before entering REM (default: 30)
dream_mind          LLM for speculation (model @ Tier, use cheaper tier)
max_cache           Maximum cached insights (default: 20)
speculation_depth   Scenarios per dream cycle (default: 3)

How It Works

Soul completes instinct cycle
  ↓
Idle for trigger_idle_sec
  ↓
DreamEngine enters REM state
  ↓
Generate scenarios from memory state:
  "What is the most likely next input?"
  "What errors might occur next?"
  "What optimization could help?"
  ↓
dream_mind LLM generates speculative insights
  ↓
Cache insights with confidence scores
  ↓
Soul wakes → sense check_dream_cache(query: "...")
  ↓ CACHE HIT
Use pre-computed result (zero primary LLM cost)

Lucid Dreaming

The check_dream_cache sense allows souls to access dream insights from their instinct block. The soul name is auto-injected by the compiler. Returns the pre-computed result if a relevant insight matches the query, or empty string if no match.

Clone Retirement

When load drops, clones die gracefully. Retirement is configured inside the mitosis block using three new fields. Clones are retired in LIFO order (last spawned = first retired) with a configurable cooldown between retirements.

Syntax

mitosis {
    max_clones: 5
    trigger: queue_depth > 10
    clone_tier: Cerebras
    retire_trigger: queue_depth < 3 && cycle_count > 10
    retire_cooldown: 120s
    min_clones: 1
}

Fields

FieldTypeDescription
retire_triggerExpressionCondition that triggers clone retirement (e.g., queue_depth < 3)
retire_cooldownDurationMinimum time between consecutive retirements
min_clonesIntegerFloor — never retire below this count

Behavior

When retire_trigger evaluates to true, the MitosisEngine retires the most recently spawned clone. After retirement, a cooldown period prevents cascading retirements. The min_clones floor ensures a minimum number of clones remain active even when load is very low.

Hot Reload

Live soul swapping with zero downtime. The --hot flag enables the HotReloadEngine, which polls the source file every 2 seconds. On change: reparse → validate → compile → diff souls → swap live.

Usage

$ nous run program.nous --hot
[hot] Watching program.nous (poll: 2s)
[hot] Change detected at 14:32:07
[hot] Reparse OK → Validate OK → Compile OK
[hot] Diff: Scout mind changed (Tier1 → Groq)
[hot] Swapping 1 soul... done (0.04s)
[hot] Live. No downtime.

How It Works

The HotReloadEngine runs alongside the main event loop. When a file change is detected:

1. The source is reparsed and validated. If validation fails, the change is rejected and the current version continues running.

2. The new AST is compiled to Python. A structural diff identifies which souls changed.

3. Only changed souls are swapped. Unchanged souls keep their state, memory, and active connections.

4. The swap is atomic — there is no moment where the system is partially updated.

Limitations

Hot reload cannot change the world structure (laws, telemetry config). World-level changes require a full restart. Soul additions and removals are supported — only the nervous system routing is rebuilt.

Reflection Cycle

Every reflect_every cycles, the ConsciousnessEngine pauses normal operation and runs an introspection cycle:

1. Measure — evaluate progress on each goal (0.0–1.0 scale).

2. Reflect — if self_model is enabled, the soul uses its LLM to analyze its own recent behavior patterns.

3. Adjust — if any goal is below goal_threshold, the soul modifies its strategy (prompt adjustments, memory prioritization, sense selection).

4. Log — reflection results are recorded in telemetry (if enabled) and memory.
